# Knowledge Base

The local knowledge base lets you ingest local files into `knowledge_documents` and `knowledge_chunks`, search them from CLI or API, and synthesize workspace findings back into reusable knowledge.

When you run `kb learn` with `scope=public`, the learned documents are now stored in the shared `public` layer while preserving the original source workspace in metadata. Keyword and vector search also default to `workspace + public` layered recall whenever a primary workspace is provided.

For semantic retrieval, Osmedeus now also supports a standalone vector SQLite database, defaulting to `{{base_folder}}/knowledge/vector-kb.sqlite`. This keeps vector embeddings portable and reusable across workflows without mixing them into the main relational DB tables.

It is also consumed by `superdomain-extensive-ai-optimized`, `superdomain-extensive-ai-stable`, `superdomain-extensive-ai-hybrid`, and `superdomain-extensive-ai-lite` during semantic search.

## Supported Document Types

- `txt`, `md`, `markdown`, `log`
- `yaml`, `yml`, `csv`
- `json`, `jsonl`
- `html`, `htm`
- `epub`
- `doc`, `docx`
- `pdf`
- `pptx`, `xlsx`

## Local Dependencies

- `docling`
  - Required for `pdf`, `docx`, `pptx`, and `xlsx`
- `antiword`
  - Required for legacy `.doc`

## Optional Dependencies

- at least one configured `llm.llm_providers` entry with an OpenAI-compatible `/embeddings` endpoint
  - Enables vector knowledge indexing/search and AI workflow semantic retrieval
- a matching `knowledge_vector.default_provider` / `knowledge_vector.default_model`
  - Keeps vectorkb index/search bound to the same provider/model pair unless you override them explicitly

## Vector Knowledge Config

```yaml
knowledge_vector:
  enabled: true
  db_path: "{{base_folder}}/knowledge/vector-kb.sqlite"
  default_provider: openai
  default_model: text-embedding-3-small
  auto_index_on_ingest: true
  auto_index_on_learn: true
  top_k: 20
  hybrid_weight: 0.7
  keyword_weight: 0.3
  batch_size: 32
  max_indexing_chunks: 5000
```

When `auto_index_on_ingest` or `auto_index_on_learn` is enabled, successful `kb ingest` and `kb learn` operations automatically refresh the standalone vector DB.

## Ingest Knowledge

Ingest a local file or directory tree into the knowledge base.

```bash
curl -X POST http://localhost:8002/osm/api/knowledge/ingest \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "path": "/data/kb",
    "workspace": "example.com",
    "recursive": true
  }'
```

**Request Body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `path` | string | Yes | Local file or directory path to ingest |
| `workspace` | string | No | Workspace-scoped knowledge namespace |
| `recursive` | bool | No | Recurse into subdirectories when `path` is a directory |

**Response**

```json
{
  "message": "Knowledge ingestion completed",
  "data": {
    "workspace": "example.com",
    "root_path": "/data/kb",
    "documents": 12,
    "chunks": 87,
    "skipped": 1,
    "failed": 0
  }
}
```

```bash
  -H "Authorization: Bearer $TOKEN"
```

**Query Parameters**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `workspace` | string | - | Filter by workspace name |
| `offset` | int | 0 | Number of documents to skip |

## Search Knowledge

Search across stored chunks using the built-in keyword scorer.

```bash
curl -X POST http://localhost:8002/osm/api/knowledge/search \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "workspace": "example.com",
    "workspace_layers": ["example.com", "shared-websec", "global"],
    "scope_layers": ["workspace", "public"],
    "query": "authentication bypass",
    "limit": 10
  }'
```

**Request Body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `workspace` | string | No | Restrict search to a workspace |
| `workspace_layers` | string[] | No | Search multiple workspaces in order of priority |
| `scope_layers` | string[] | No | Prefer matching knowledge scopes such as `workspace`, `project`, `public` |
| `query` | string | Yes | Search phrase |
| `limit` | int | No | Maximum hits to return |

**Response**

```json
{
  "query": "authentication bypass",
  "workspace": "example.com",
  "total": 2,
  "data": [
    {
      "document_id": 11,
      "chunk_id": 47,
      "workspace": "example.com",
      "title": "JWT Playbook",
      "source_path": "/data/kb/jwt-playbook.md",
      "doc_type": "md",
      "section": "Bypass Notes",
      "snippet": "Try alg confusion and key confusion around JWT verification...",
      "score": 32.0,
      "metadata": {
        "scope": "public",
        "knowledge_layer": "public",
        "source_workspace": "example.com",
        "sample_type": "verified",
        "source_confidence": 0.95,
        "target_types": ["url"],
        "labels": ["auto-learn", "verified"]
      }
    }
  ]
}
```

## Learn Knowledge from a Workspace

Generate structured learned knowledge documents from existing workspace assets, vulnerabilities, runs, and optional `ai-analysis` artifacts.

```bash
curl -X POST http://localhost:8002/osm/api/knowledge/learn \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "workspace": "example.com",
    "scope": "workspace",
    "max_assets": 20,
    "max_vulnerabilities": 20,
    "max_runs": 10,
    "include_ai_analysis": true
  }'
```

**Request Body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `workspace` | string | Yes | Source workspace name |
| `scope` | string | No | Summary scope: `workspace`, `project`, or `public` |
| `max_assets` | int | No | Maximum assets to include |
| `max_vulnerabilities` | int | No | Maximum vulnerabilities to include |
| `max_runs` | int | No | Maximum runs to include |
| `include_ai_analysis` | bool | No | Include files from `<workspace>/ai-analysis` |

**Response notes**

- `workspace` remains the source workspace that was learned from.
- `storage_workspace` reports where the learned documents were actually stored. For `scope=public`, this will be `public`.

## Index into the Standalone Vector DB

Trigger indexing from the main knowledge tables into `vector-kb.sqlite`.

```bash
curl -X POST http://localhost:8002/osm/api/knowledge/vector/index \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "workspace": "example.com",
    "provider": "openai",
    "model": "text-embedding-3-small",
    "max_chunks": 500
  }'
```

**Request Body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `workspace` | string | No | Restrict indexing to a single workspace; empty indexes all workspaces |
| `provider` | string | No | Override configured embedding provider |
| `model` | string | No | Override configured embedding model |
| `max_chunks` | int | No | Maximum chunks to index in this run |

```bash
    "scope_layers": ["workspace", "public"],
    "query": "authentication bypass",
    "limit": 10
  }'
```

**Request Body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `workspace` | string | No | Restrict search to a workspace |
| `workspace_layers` | string[] | No | Search multiple workspaces in order of priority |
| `scope_layers` | string[] | No | Prefer matching knowledge scopes such as `workspace`, `project`, `public` |
| `query` | string | Yes | Natural-language semantic query |
| `provider` | string | No | Override configured embedding provider |
| `model` | string | No | Override configured embedding model |

## Get Vector DB Stats

```bash
curl http://localhost:8002/osm/api/knowledge/vector/stats \
  -H "Authorization: Bearer $TOKEN"
```

This returns the vector DB path, document count, chunk count, embedding count, indexed workspaces, and active provider/model combinations.

## Workflow Integration

For the AI workflow family, the default knowledge workspace is `{{TargetSpace}}`.

Recommended flow:

1. Ingest your PDFs, markdown notes, ebooks, or vendor docs into the same workspace name you scan with.
2. Let Osmedeus auto-index them into `vector-kb.sqlite`, or trigger `/knowledge/vector/index` manually.
3. Run `superdomain-extensive-ai-hybrid`, `superdomain-extensive-ai-stable`, `superdomain-extensive-ai-optimized`, or `superdomain-extensive-ai-lite`.
4. The workflow will perform direct `kb vector search`, merge that with direct `kb search` keyword hits, still export chunks into the semantic corpus, and run post-report `kb learn` for reuse in future runs.
5. During the AI closure, the workflow also writes a normalized `applied-ai-decision-<workspace>.json`, feeds targeted rescan findings back into the main vuln result set, and emits a `followup-decision-<workspace>.json` package after retest/operator/campaign coordination.
