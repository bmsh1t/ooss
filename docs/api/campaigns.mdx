# Campaigns

Campaigns group multiple queued runs under one shared strategy, role, notes, and optional deep-scan policy.

Campaign target aggregation now includes both:

- `vuln_summary`
- `attack_chain_summary`

That means a target can be promoted to high-risk not only by critical/high vulnerabilities, but also by attack-chain signals in the same workspace. For campaign risk decisions, Osmedeus keeps `retest`-stage linked chains in the operational window until they are disproven or closed.

## List Campaigns

```bash
curl http://localhost:8002/osm/api/campaigns \
  -H "Authorization: Bearer $TOKEN"
```

## Create a Campaign

```bash
curl -X POST http://localhost:8002/osm/api/campaigns \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "wide-recon-batch",
    "flow": "general",
    "targets": ["example.com", "api.example.com"],
    "priority": "high",
    "auto_deep_scan": true,
    "deep_scan_workflow": "superdomain-extensive-ai-hybrid",
    "high_risk_severities": ["critical", "high"]
  }'
```

Important request fields:

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Campaign name |
| `flow` / `module` | string | Workflow to queue |
| `targets` | array | Inline targets |
| `target_file` | string | File path containing targets |
| `priority` | string | `low`, `normal`, `high`, `critical` |
| `auto_deep_scan` | bool | Automatically queue a deep-scan run when high-risk findings appear |
| `deep_scan_workflow` | string | Follow-up workflow name |
| `deep_scan_workflow_kind` | string | `flow` or `module` |
| `high_risk_severities` | array | Severities that trigger deep scan |

## Get Campaign Status

```bash
curl http://localhost:8002/osm/api/campaigns/<campaign-id> \
  -H "Authorization: Bearer $TOKEN"
```

The status response includes per-target aggregation fields such as:

- `risk_level`
- `vuln_summary`
- `attack_chain_summary`
- `deep_scan_queued`

It also includes a top-level `summary` block with aggregate counters such as:

- `targets_total`
- `high_risk_targets`
- `deep_scan_queued_targets`
- `attack_chain_aware_targets`
- `verified_attack_chain_targets`

`attack_chain_summary` contains rolled-up values like:

- `report_count`
- `total_chains`
- `critical_chains`
- `high_impact_chains`
- `queue_hits`
- `verified_hits`
- `operational_hits`
- `last_queued_at`

## Get Campaign Report

```bash
curl http://localhost:8002/osm/api/campaigns/<campaign-id>/report \
  -H "Authorization: Bearer $TOKEN"
```

The report endpoint keeps the status response and adds campaign-level analytics:

- `risk_distribution`
- `latest_run_status_distribution`
- `trigger_distribution`
- `deep_scan`
- `rerun_history`
- `targets`

Optional query filters narrow the returned `targets` slice while keeping the top-level campaign analytics as full-batch totals:

| Query | Example | Purpose |
|-------|---------|---------|
| `risk` | `risk=high,critical` | Match target `risk_level` |
| `status` | `status=completed,failed` | Match target latest run status |
| `trigger` | `trigger=campaign-rerun` | Match target latest trigger type |
| `preset` | `preset=high-risk` | Apply an operator preset |
| `offset` | `offset=20` | Skip N matched target rows after filters |
| `limit` | `limit=50` | Return at most N matched rows after filters (`0` = all) |
| `sort_by` | `sort_by=target` | Sort by `risk`, `target`, `latest_run`, or `open_high_risk` |
| `sort_order` | `sort_order=asc` | Sort order override (`asc` or `desc`) |
| `profile` | `profile=ops-handoff` | Apply a saved report/export profile before explicit query overrides |

Available presets:

- `high-risk`
- `recovered`
- `failed`

The response also includes:

- `filters_applied`
- `sort_applied`
- `total_targets`
- `result_count`
- `pagination`

`result_count` is the number of matched rows after filters. `pagination.returned_count` is the number of rows returned in the current page.

Each `targets` row is a flattened audit view with:

- latest non-deep-scan run metadata
- rerun count and recovery flag
- deep-scan count and queued state
- per-target finding counts
- attack-chain `operational_hits` and `verified_hits`

`deep_scan` helps measure follow-up conversion:

- `eligible_targets`
- `queued_targets`
- `total_runs`
- `queued_runs`
- `running_runs`
- `completed_runs`
- `failed_runs`
- `conversion_rate`

`rerun_history` summarizes recovery behavior:

- `total_runs`
- `unique_targets`
- `recovered_targets`
- `last_rerun_at`

## Export Campaign Report

```bash
curl http://localhost:8002/osm/api/campaigns/<campaign-id>/export \
  -H "Authorization: Bearer $TOKEN" \
  -o campaign-report.csv
```

Default export format is CSV. To export the same payload as JSON instead:

```bash
curl "http://localhost:8002/osm/api/campaigns/<campaign-id>/export?format=json" \
  -H "Authorization: Bearer $TOKEN"
```

You can combine export with the same row filters and presets:

```bash
curl "http://localhost:8002/osm/api/campaigns/<campaign-id>/export?preset=failed" \
  -H "Authorization: Bearer $TOKEN" \
  -o failed-targets.csv
```

Pagination works the same way for export:

```bash
curl "http://localhost:8002/osm/api/campaigns/<campaign-id>/export?status=completed&offset=100&limit=50" \
  -H "Authorization: Bearer $TOKEN" \
  -o campaign-page.csv
```

Sorting works the same way for export:

```bash
curl "http://localhost:8002/osm/api/campaigns/<campaign-id>/export?sort_by=target&sort_order=desc" \
  -H "Authorization: Bearer $TOKEN" \
  -o campaign-sorted.csv
```

Saved profiles can also drive both report and export:

```bash
curl "http://localhost:8002/osm/api/campaigns/<campaign-id>/report?profile=ops-handoff" \
  -H "Authorization: Bearer $TOKEN"

curl "http://localhost:8002/osm/api/campaigns/<campaign-id>/export?profile=ops-handoff" \
  -H "Authorization: Bearer $TOKEN"
```

When a profile is applied:

- saved `filters` and `sort` are loaded first
- saved `format` becomes the default export format
- explicit query parameters still win over the saved profile

## Saved Report Profiles

List saved profiles:

```bash
curl http://localhost:8002/osm/api/campaigns/<campaign-id>/profiles \
  -H "Authorization: Bearer $TOKEN"
```

Save or update a reusable profile:

```bash
curl -X PUT http://localhost:8002/osm/api/campaigns/<campaign-id>/profiles/ops-handoff \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "description": "operator handoff view",
    "filters": {
      "statuses": ["completed"],
      "trigger_types": ["campaign-rerun"]
    },
    "sort": {
      "by": "target",
      "order": "desc"
    },
    "format": "json"
  }'
```

Delete a saved profile:

```bash
curl -X DELETE http://localhost:8002/osm/api/campaigns/<campaign-id>/profiles/ops-handoff \
  -H "Authorization: Bearer $TOKEN"
```


## Queue Deep Scan for High-Risk Targets

```bash
curl -X POST http://localhost:8002/osm/api/campaigns/<campaign-id>/deep-scan \
  -H "Authorization: Bearer $TOKEN"
```

Deep-scan target selection now treats a workspace as high-risk when either of these is true:

- it has vulnerabilities matching `high_risk_severities`
- it has operational attack-chain signals that match the same risk band

In practice:

- `critical` risk can be triggered by verified or `retest` critical chains
- `high` risk can be triggered by verified or `retest` high-impact chains
